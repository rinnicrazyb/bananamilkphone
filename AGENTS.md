# 香蕉牛奶机 · Agent 协作规则

> 本文档与《bananamilkphone项目书》和开发日志配套使用。项目书回答"最初的静态内容"，开发日志回答"动态的演变"，本文档回答"怎么保证做对、怎么防止走样"。新会话开始必须三份都读，缺一不可。
> 最后更新：2026-07-16

## 0. 新会话启动协议

任何新会话、任何 agent 接手这个项目，第一步必须完成、且必须在开始任何代码修改之前完成：

1. 完整读取开发日志全部内容（或至少最近一个 Phase 以来的全部记录）
2. 完整读取项目书当前版本
3. 若存在已归档的技术预研报告（见第 1 条），一并读取
4. 简要向用户复述当前项目所处阶段、已完成的 APP、已确认的关键技术决策，确认理解无误后再开始工作

禁止跳过 1-3 直接进入代码修改。日志和项目书是写给"未来接手的 agent"看的，没有强制先读这一步，日志就只是写了没人看的仪式，新 agent 还是从零理解项目，甚至可能跟已经拍过板的技术判断打架。

---

## 1. 技术方案与预研报告机制

用户不写代码，全靠 agent 判断技术可行性，所以技术栈相关内容 agent 必须谨慎对待：

- 涉及技术选型时，agent 必须做正反分析：解释技术原理、说明项目目标功能的可实现性、给出替代方案（含利弊），用不懂代码的人也能大致判断的方式表达。
- **触发条件**：若项目书中描述的某项功能，在当前技术栈下实现成本极高，或触及系统权限壁垒（例如需要用户手动授予敏感权限、可能撞上应用商店审核政策等）→ 必须停下来，不得直接开工，生成一份"技术预研报告"。
- **报告格式**：问题描述 + 方案 A（含利弊）+ 方案 B（含利弊）+ agent 的推荐意见（可以有倾向，但要说明理由）。
- 报告发给用户决策：维持原方案、改用替代交互、或延后至后续阶段（如原生插件开发阶段）。
- **报告确认后必须回写项目书**：用户选定方案后，agent 必须同步修改项目书对应章节，把最终方案写回去；技术预研报告本身归档留存，但不能是"报告说了算、项目书没更新"的状态。半年后新 agent 接手，读到的项目书必须是已经吸收过决策结果的最新版本，不能让它把已经拍过板的问题重新纠结一遍。

---

## 2. 决策分级与询问规则

不是所有不确定的地方都需要打断用户确认。全问会导致真正重要的问题被大量琐碎提问淹没；不问又会让 agent 悄悄替用户做决定。分两档处理：

**必须问，问到摸清真实需求为止：**
- 会改变用户能感知到的行为/体验的决定（交互方式、某功能保留与否、数据存储/删除方式）
- 任何计划外的想法、超出项目书范围的建议
- 任何功能模块的取舍——不得擅自砍掉或跳过功能模块

**agent 自行决定即可，开发日志注明假设，不必每次打断：**
- 纯实现细节、用户完全无感知差异的选择（变量命名、内部函数怎么拆分、样式具体数值只要视觉上没有明显偏离项目书描述）

**grill-me 技能的使用场景：**
- 用在每个新 APP、每个大功能正式开发启动前，做一次系统性、结构化的访谈，把该功能的细节、边界情况、跨 APP 耦合关系一次性摸清，产出这个 APP 开发阶段的详细规格（TASK_SPEC）
- 不用在开发过程中零散的小问题上——那些走上面"决策分级"规则处理，避免高频碎片化提问打断开发节奏，也避免访谈式的深度模式被用在不需要那么重的问题上

## 4. 代码改动流程（按"爆炸半径"分级）

不是每次改动都要走最重的全套流程。按改动影响范围分三级：

**共享服务级**（事件总线、存储封装、类型定义、跨 APP 共享的 Context/Store）→ 走全套：
1. 改前用搜索工具扫描所有引用该函数/变量/类型的文件，生成"影响面清单"
2. 判断修改方案，评估是否会在耦合度高的地方引入 bug，写入开发日志
3. 精准修改需要改的部分，不得整份文件重写；非 UI 逻辑（API 调用、数据处理）必须同步补充/更新对应单元测试
4. 改完立刻回归测试，逐项核对影响面清单，有 bug 立即修复并重新测试，直到无 bug
5. 用户反馈bug多次修复仍存在的情况，考虑是否技术栈能力硬性制约，分析用户目标的最佳实现技术栈方案，询问用户是否引入新技术，不要死磕bug原地打转。

**单 APP 内部级**（改动不涉及对外接口变化，不影响其他 APP）→ 精简流程：
- 本地测试 + 该 APP 内部回归即可，不需要全项目 grep 扫描

**样式/文案级**（纯视觉、纯文字调整，无逻辑变化）→ 最简流程：
- 改完肉眼确认一遍即可

不整份文件重写这条原则适用于所有级别，不因分级而放松。

## 5. 测试与验收标准

- **测试框架**：Phase 1 阶段确定引入 Vitest（与 Vite 共用配置，接入成本低），非 UI 逻辑的单元测试基于此编写。
- **阶段交付验收**：每完成一个 Phase/APP，做一次该 APP 完整回归测试；每完成一处跨 APP 耦合代码，做一次联调测试；测试通过后生成验收清单（Checklist，覆盖该 APP 所有交互路径），发给用户，用户在浏览器/真机上体验并反馈。
- **每轮交付汇报格式**（按改动分级调整篇幅，共享服务级改动需完整回答，单 APP/样式级可精简）：
  - 本轮行动目标
  - 若修 bug：根因分析；若开发功能：功能效果 + 耦合对象清单
  - 技术选型说明：用了什么技术、为什么选它、极端情况下是否有隐患
  - 耦合代码/耦合功能是否已同步修改并确认对接正常
  - 回归测试结果（功能单元测试 / 所在 APP 测试 / 耦合测试 / 电脑浏览器测试 / apk 打包测试）

**回归测试硬性标准：**
- 禁止仅凭终端无报错就宣称"验证通过"
- 必须明确列出本次修改波及的 UI 组件清单（上游调用方 + 下游被调用方）
- 耦合联动检查：修改了公共函数、全局状态（Context/Store）或数据接口（TypeScript Interface）时，必须用 grep/代码搜索工具扫描全项目所有引用处，逐一检查调用处传参和取值是否仍匹配
- 双端测试：电脑浏览器 + 安卓 apk；若无法正常打包成 apk，必须查明原因并禀报用户

---

## 6. 定期架构复查

耦合检查解决的是"这次改动有没有漏改关联代码"，这是局部正确性问题。但有一类问题不是哪次改动"漏改"了什么——每次改动单独看都没错，攒起来才会出现设计层面不咬合的地方（比如两套互相不知道对方存在的机制、同一个功能在不同章节的规格对不上）。局部耦合检查抓不住这种问题。

- 每完成 2-3 个 APP，做一次跳出单次改动的整体架构复查：检查是否有功能重复、职责不清、多套机制互不感知的情况，检查项目书当前版本内部是否自洽
- 复查发现的问题按第 1 条"技术预研报告"机制处理：出报告、用户确认方案、回写项目书
- 这与第 5 条的阶段验收测试是两个维度，不能互相替代——验收测试测的是"功能对不对"，架构复查看的是"设计还合不合理"
