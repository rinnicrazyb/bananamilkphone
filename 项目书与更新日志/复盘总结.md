# 复盘总结 — 2026-07-21

> 为什么四次修复后仍有大量 bug，以及如何根治。

---

## 一、核心问题：数据结构没有对齐参考项目

### 问题描述

RikkaHub 的消息数据结构是 `messageNodes: List<MessageNode>`，每个节点是一个"槽位"，槽位顺序永不改变，分支是在槽位内部选择不同版本。

我们四次修复都在 flat `Message[]` 数组上打补丁：先加 `nodeId` 字段，再加 `nodeSelections`，再加 `getVisibleMessages` 排序逻辑……每次都是"让 flat 数组表现得像节点树"，但每次都有边界情况漏掉。

### 教训

**"部分对齐"比"不对齐"更危险**。部分对齐让你以为问题解决了，但边界情况（切换分支后排序错乱、编辑后内容不更新等）会不断出现，每次都要修一次。

**正确做法**：要么不改数据模型，每次手动兼容；要么一次性改成完全一样的数据模型，不要做中间过渡方案。

### 如何判断应该全改还是打补丁

| 情况 | 应该 |
|------|------|
| 改动影响 1 个函数/1 个文件 | 打补丁 |
| 改动影响 3 个以上的模块（store、渲染、发送、持久化） | 全改 |
| 需要新增"兼容层"来弥合差异 | 全改，兼容层本身就是 bug 来源 |
| 参考项目用这套模型稳定运行了多年 | 全改，不要相信自己能发明更好的 |

## 二、具体失误清单

### 1. 流式回调只写 `messages` 兼容字段，不写 `messageNodes` 主结构

**发生原因**：改 store 数据结构时，保留了 `messages` 兼容字段让旧代码继续工作。但 `chat-send/index.ts` 的流式回调、工具更新、错误处理、已读标记——4 处代码都在读写兼容字段，不是主结构。

**修复方式**：逐一 grep 所有 `store.messages[` 引用，全部改为 `store.messageNodes` + `updateNodeMsg` 工具函数。

**预防措施**：改数据模型后立刻用 `grep` 扫描所有旧字段引用，列影响面清单。不要假设旧代码会自动适配。

### 2. ContextPreviewPage 在拿到正确顺序后又被 `sort(timestamp)` 重排

**发生原因**：`getCurrentMessages` 已经按节点顺序返回正确结果（对齐 RikkaHub），但旧代码在第 97 行又按时间戳排序了。这个 `.sort()` 在旧 flat 模型下是必需的（flat 模型没有节点，只能按时间戳排序），但模型改后变成了破坏者。

**修复方式**：删除第 97 行的 `sort`。

**预防措施**：改数据模型后，逐一审查所有数据处理代码，判断哪些逻辑是基于旧模型的假设写的，是否需要删除或修改。

### 3. CopyButton 用 `useEffect` + DOM append 被 React 覆盖

**发生原因**：React 的 Virtual DOM 在每次 re-render 时对比真实 DOM 和虚拟 DOM。`useEffect` 添加的 DOM 元素不在虚拟 DOM 中 → React 认为"多余" → 删除。这个机制在 TauriTavern（vanilla JS）中不存在。

**修复方式**：改为 JSX 渲染 `<CopyButton>` 组件，成为虚拟 DOM 的一部分。

**预防措施**：在 React 项目中，所有 UI 元素都应通过 JSX 渲染。只在以下情况用 `useEffect` + DOM 操作：
- 第三方库集成（video.js、chart.js）
- 需要直接操作 CSSOM（如 iframe 高度、canvas）
- 上述情况必须用 ref，确保 React 不管理该属性

### 4. 四次修复都是在同一层次打补丁，没有挖到根因

**发生经过**：
- 第一次：加 `nodeId` 字段
- 第二次：加 `nodeSelections` + `getVisibleMessages`
- 第三次：`editMessageContent` 清除 `parts`
- 第四次：加 `excludeNodeIds`
- 第五次（终于）：改 `messageNodes` 数据结构

**教训**：如果同一个问题修了两次还没好，说明根因不在你改的那一层。应该停下来做影响面分析，而不是继续在同一层打补丁。

### 5. 没有遵守 AGENTS.md 的 "影响面清单" 流程

AGENTS.md 第 4 条要求共享服务级改动前用 `grep` 扫描所有引用，生成"影响面清单"。这次改 `messages` → `messageNodes` 时没有做这一步，导致 5 处旧引用漏改。

**预防措施**：涉及 store 字段改名、函数签名变化时，强制 grep + 逐文件检查。

## 三、与参考项目 RikkaHub 的关键差异

| 方面 | RikkaHub | 我们学到了 |
|------|----------|-----------|
| 数据结构 | `messageNodes: List<MessageNode>` | 节点位置由数组 index 固定，永不重排 |
| currentMessages | `nodes.map { it.messages[it.selectIndex] }` | 显示和 LLM 上下文用同一个推导 |
| 编辑消息 | 追加新 UIMessage 到 node.messages，selectIndex++ | 不修改原消息，保留完整历史 |
| 重新生成 | `messageRange` 排除旧回复 | 不 truncate store，用 excludeNodeIds |
| 流式持久化 | `saveConversation()` 写完整 Conversation | 流式结束后 `dbUpdateMessage` 写单条消息 |
| 上下文预览 | 不存在（RikkaHub 无此功能） | 我们做了但需确保与 LLM 上下文一致 |

## 四、代码修改原则（补充到 AGENTS.md）

基于本次教训，建议补充以下原则：

### 4.1 数据模型变更流程

1. 新类型定义 → 2. 改 store（主结构）→ 3. grep 全项目旧字段引用 → 4. 逐文件更新 → 5. 删兼容层 → 6. 编译验证 → 7. 测试每个受影响功能

### 4.2 兼容字段红线

- 兼容字段只能用于"读取"（旧组件过渡期用）
- 兼容字段不能用于"写入"——任何写操作必须通过主结构
- 兼容字段应在所有旧消费者迁移后立即删除
- 兼容字段的存活时间不应超过 1 次提交

### 4.3 React DOM 操作规则

- UI 元素：必须通过 JSX 渲染
- 第三方集成：用 ref 直接操作，不在 JSX 中声明 ref 管理的属性
- 高度/尺寸变更：用 ref 直接操作 style，不在 state 中管理
- 如有例外：注释写明原因 + 标注 "React 不管理此属性"
