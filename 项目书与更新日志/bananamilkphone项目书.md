## 一、项目定位

软件名称：**香蕉牛奶机（Banana Milk Phone）**，属于现今流行的「小手机」项目——一个融合人机恋聊天与记忆系统的 AI 伴侣安卓软件。

特色：
- 软件设计犹如一部真正的手机，打开香蕉牛奶机如同打开一部手机，手机桌面布局，4×6APP排序，小手机APP，桌面/锁屏壁纸，桌面小组件，系统监听等像真正的手机一样！
- 小手机的目标是让 AI 伴侣能真正**参与用户的生活**——时间感知、事件感知、跨设备连续——像人一样陪在身边。
- 虚拟与真实的联动：在小手机里AI可以感知到用户真实手机的使用情况，时间、天气、应用使用情况、主动发消息，同时也可以依赖小手机内部的事件监听，如打开小手机内部的街机厅软件——AI收到监听推送——弹窗发消息/锁应用等……

## 二、技术栈（暂定，如功能实现需要其他更便捷的，协商会审后可更改）

| 层级      | 技术选型                        | 说明                                              |
| ------- | --------------------------- | ----------------------------------------------- |
| 前端框架    | **React 18 + TypeScript**   | 组件化，类型安全                                        |
| 构建工具    | **Vite**                    | 快速 HMR                                          |
| 跨平台壳    | **Capacitor**               | 打包 Android APK；webDir→dist, androidScheme→https |
| 本地存储（主） | **IndexedDB**               | 记忆系统 + 聊天记录 + 主题配置 + 设置参数等应用内配置                 |
| 备份      | **本地备份+WebDAV（坚果云）**        | 坚果云（如有其他技术或协议要求，比如必须原生安卓，则优先满足其技术性要求）           |
| APP 通信  | **事件总线（IndexedDB 持久化）**     | APP 间低耦合，事件驱动，不直接引用                             |
| 主题控制    | **CSS 变量**                  | 主题引擎独立，只控制 CSS 变量，不干涉 APP 内部                    |
| AI 调用   | **LLM API + Tool Call**     | 依赖 Tool Call 实现APP间的联动                          |
| 状态管理    | **React Context / Zustand** | 轻量                                              |
| 图标      | **Phosphor Icons**          | 专业线条图标库                                         |
| 响应式设计   |                             | 所有界面及UI自适应用户屏幕大小，适配手机端、电脑端、平板端，                 |

### 关键架构决策

1. **Prompt可视化** — 所有APP中都添加预览功能，可选择条件设定后预览完整发送给llm的文本拼装。
2. **prompt自定义**——所有功能的提示词都显性设计，用户可使用默认提示词或自定义内容。
3. **工具命名"像呼吸一样自然"** — 通过生命哲学设计，让AI在调用工具时如人张嘴吃饭般自然，将工具自然融入到意识中唤醒其伴侣身份而非AI身份
4. **精准修改，不整份重写** — 改代码前 build 确认旧功能正常→精准定位修改→改完验证→回归测试，不整份重写，不重复白干……确认需求和目标，选择手段，精准定位，改完验证，回归测试。

### 3.2 命名规范

- **组件**：PascalCase — `ChatMessage.tsx`
- **文件/目录**：kebab-case — `message-list.tsx`
- **函数/变量**：camelCase
- **常量**：UPPER_SNAKE_CASE
- **CSS 变量**：`--app-xxx-yyy`
- **事件名**：`app-name:event-name` — `chat:message-sent`

### 3.3 组件规范

- 每个 APP 下含 `pages/`、`components/`、`hooks/`、`store/` 子目录
- APP 间不直接引用组件，通过事件总线通信
- 共享 UI 组件放顶层 `components/`
- 每个 APP 独立路由前缀（如 `/chat`、`/memory-gallery`）

### 3.4 精准修改原则

1. 改前先 build 确认旧功能正常
2. 用 grep_code + grep_context 定位目标代码，理解上下文后再改
3. 只改需要改的函数/组件/样式，不得整份文件重写
4. 改完立刻 build + 手动测试受影响功能
5. 每完成一个 APP 做一次全量回归测试

### 3.5 工具命名哲学（需后续商讨）

- 用伴侣视角写tool call工具是干嘛的、什么时候该用，像呼吸一样自然

### 3.6 Git 提交规范

```
<type>(<scope>): <简短描述>

类型: feat / fix / refactor / style / docs / chore
示例:
  feat(chat): 添加消息已读状态
  fix(memory-gallery): 修复搜索时翻页重置
```

#### 2-6  世界书 APP（lorebook APP）

**技术设计**：完全参照 RikkaHub 开源项目的 Lorebook 系统（PromptInjectionTransformer 注入机制），UI 上做书架书封美化。

**数据模型**：
- **世界书（Lorebook）**：容器，含名称、描述、书封（可选）、启用/禁用、多条条目
- **条目（LorebookEntry）**：单条注入规则，每个条目独立配置关键词、注入内容、注入位置、优先级

**注入位置（5 种，来源于 RikkaHub）**：
| 位置 | 效果说明 |
|------|---------|
| 系统提示词前（BEFORE_SYSTEM_PROMPT） | 内容合并到 system message 最开头 |
| 系统提示词后（AFTER_SYSTEM_PROMPT） | 内容合并到 system message 末尾（最常用） |
| 对话开头（TOP_OF_CHAT） | 在第一条用户消息前插入独立消息（可指定 USER/ASSISTANT 角色） |
| 最新消息前（BOTTOM_OF_CHAT） | 在最后一条消息前插入独立消息 |
| 指定深度（AT_DEPTH） | 从最新消息往前数 N 条位置插入独立消息 |

注：前 2 种为"修改系统提示词"，后 3 种为"插入独立消息"——插入独立消息时可以指定该消息扮演 USER 还是 ASSISTANT 角色。

**激活机制**：
- **常驻激活**：无论对话内容如何，始终注入到设定位置
- **关键词激活**：仅在最近 N 条消息中出现关键词时触发注入
- 支持普通关键词和正则表达式两种匹配模式

**核心交互**：
- 匹配时扫描最近 N 条消息（scanDepth，用户可配置，默认 5 条）
- 优先级 1-100 控制同一位置多条注入的排列顺序
- 同位置同优先级按条目列表顺序排列

**UI布局（书籍阅读式体验）**：
- **书架**：网格布局，一行 2 本书，显示书封+名称+简介，按编辑时间排序
- **封面页**（第 1 页）：书封大图 + 名称 + 简介 + 启用开关 + 绑定智能体信息（只读）
- **目录页**（第 2 页）：条目列表，每行显示条目名+关键词预览+启用状态
- **条目页**（第 3 页起）：每个条目一页，显示完整信息（名称、关键词、注入位置、优先级、注入内容正文等）
- **翻页**：手机左右滑动 + 电脑底部左右箭头双支持
- **侧边栏**：从屏幕左边缘向右滑出，显示所有条目名称列表+启用状态，点击跳转

互动功能

#### 3-2  街机厅 APP（Arcade APP）——含沙盒游戏引擎

**定位：** 小游戏平台 + 自定义游戏导入沙盒。（不是固定游戏列表，而是可扩展的游戏平台。），实现用户和智能体一起玩游戏和智能体自行独立玩游戏

**核心设计：**

```
街机厅 APP
├── 内置游戏
│   ├── 五子棋（人机对战）
│   └── 象棋（人机对战）
├── 游戏引擎层
│   ├── 渲染接口（Canvas 2D）
│   ├── 输入接口（触摸/键盘）
│   ├── 生命周期（init → update → render → destroy）
│   └── 状态管理（游戏内数据隔离）
├── 游戏注册表（IndexedDB 存储已安装游戏列表）
├── 导入模块（解析 zip → 验证 → 注册）
└── API 规范文档（APP 内可查阅）
```

**沙盒游戏导入流程：**

```
1. AI可阅读 APP 内的游戏 API 规范文档
2. 按规范编写游戏代码（manifest.json + game.js + assets/）
3. 打包为 zip 文件
4. 用户导入 zip → APP 解析：
   a. 校验 manifest.json（name/version/入口文件）
   b. 沙箱环境加载 game.js（iframe + postMessage 或 Web Worker）
   c. 注册到游戏列表
5. 用户从街机厅启动游戏
```

**UI 布局：**

```

│ ← 街机厅          📖 开发规范      │
├────────────────────────────────────┤

│ ┌──────┐  ┌──────┐               │
│ │ ⚫⚪  │  │ ♜♝   │               │
│ │ 五子棋│  │ 象棋  │               │
│ │ 3胜2负│  │ 1胜0负│               │
│ └──────┘  └──────┘               │
│                                    │
│ 已导入                            │
│ ┌──────┐  ┌──────┐               │
│ │ 💣   │  │ 🏓   │               │
│ │ 扫雷  │  │ 打砖块│               │
│ │ 7胜3负│  │ 最高分:500│          │
│ └──────┘  └──────┘               │
│                                    │
│ [📥 导入游戏]                      │
└────────────────────────────────────┘
```

**互动功能：**

| 功能       | 说明                          | 实现                                |
| -------- | --------------------------- | --------------------------------- |
| 内置游戏     | 五子棋、象棋（人机对战）                | 内置代码                              |
| 游戏导入     | 导入 zip 添加新游戏                | zip 解析 + 沙盒加载                     |

**注意事项：**

- 导入时校验 manifest 格式和入口文件存在性
- 游戏代码仅能通过暴露的 API 与外界交互（Canvas 渲染/输入事件）

**最终成效：** 用户不仅能玩内置游戏，还能让AI按规范写新游戏 → 打包导入 → 即时可玩。街机厅变成了可无限扩展的游戏平台，本质是提前搭建了一套插件系统。

## 五、开发全流程注意事项

### 5.1 旧功能保护（最高优先级）
- 改前 build 确认当前功能正常
- 精准定位修改，**不得整份文件重写**
- 改完立即 build + 手动测试受影响功能
- 每完成一个 APP 做全量回归测试

### 5.2 屏幕适配与手机交互
- 自适应电脑、平板、手机尺寸
- 手机手势交互
- 做好每个界面的返回键

### 5.3 性能
- 聊天/记忆列表必须虚拟滚动
- 图片懒加载 + 缓存
- IndexedDB 写入防抖 + 批量
- React useEffect 清理防内存泄漏

### 5.4 数据库与迁移
- IndexedDB Schema 版本管理 + 迁移函数
- WebDAV 同步不覆盖，用户可自行选择版本
- 关键操作写日志便于排查

### 5.5 错误处理
- LLM API 调用 try-catch，用户可见友好提示
- 网络错误指数退避重试（最多 3 次）
- WebDAV 失败不阻塞 APP 使用

### 5.6 加密安全
- API Key / WebDAV 密码使用 Web Crypto API 加密
- 敏感配置不在日志明文输出
- 聊天消息可选 AES-GCM 加密

### 5.7 事件总线规范
- 事件名 `app-name:event-name`
- 类型定义在 `types/events.ts`
- 超时机制，防止死等
- 消费后标记已读，不重复消费

### 5.8 监听与推送
- 事件监听：可监听真实手机，也可以监听香蕉牛奶机内APP使用
- 弹窗：真实手机通知栏弹消息，香蕉牛奶机内弹窗在屏幕正中央

### 5.9 新增 APP 步骤
1. `src/apps/` 下创建目录
2. 桌面 APP 注册列表添加
3. 实现功能，不碰其他 APP 代码
4. 设置 APP 添加相关配置项
5. 回归测试已有 APP

### 5.10 数据保存
- 本地 IndexedDB 有 + WebDAV 云端有
